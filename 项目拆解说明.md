# 项目拆解说明

## 1. 项目定位与整体判断

该项目是一个基于 OpenCode 改造的 Web AI Agent 平台，目标不是单纯做聊天界面，而是把“模型调用、工具执行、权限控制、会话编排、文件操作、MCP 扩展、实时事件流”整合成一个可交互的 Agent 工作台。

从当前代码看，它采用了典型的前后端分离形态：

- 前端：`web/`，负责任务发起、会话展示、文档编辑、技能管理、调度管理和实时过程可视化。
- 后端：`backend/`，负责真正的 Agent Runtime，包括 Provider 抽象、Tool Registry、Session 编排、权限服务、MCP 集成、SQLite 持久化、SSE 事件推送。
- 部署层：`Dockerfile`、`docker-compose.yml`、`deploy/nginx.conf`，用于将 Web 与 Backend 组合成可部署服务。

它不是一个“页面壳子 + 简单 API”的项目，而是一个以后端为核心的大型 Agent 编排系统。

## 2. 服务运行时架构拆解

### 2.1 启动链路

服务启动入口在 `backend/src/index.ts`：

1. 初始化日志系统。
2. 处理若干环境变量兼容映射，例如 `ANTHROPIC_AUTH_TOKEN -> ANTHROPIC_API_KEY`。
3. 读取 `PORT` / `HOST`。
4. 调用 `Server.listen(...)` 启动 Hono 服务。

也就是说，后端不是脚本式 CLI，而是一个常驻 HTTP Agent Runtime。

### 2.2 请求进入后的运行方式

核心 HTTP 组装在 `backend/src/server/server.ts`，处理链路可以概括为：

1. 全局错误处理。
2. 可选 Basic Auth 校验。
3. 请求日志记录。
4. CORS 放行。
5. 解析 `directory` 或 `x-opencode-directory`。
6. 将请求绑定到对应项目实例 `Instance.provide(...)`。
7. 路由转发到 `project / session / provider / permission / mcp / document / schedule` 等子系统。

这意味着后端运行模式不是单一全局状态，而是“按项目目录隔离实例上下文”的多实例运行时。

### 2.3 实例化运行时

实例上下文在 `backend/src/project/instance.ts` 中管理，核心能力包括：

- 按目录缓存实例。
- 自动推导 `project` 和 `worktree`。
- 支持 `reload` / `dispose` / `disposeAll`。
- 为同一项目目录复用状态容器。

实例启动时会执行 `backend/src/project/bootstrap.ts`，初始化：

- Plugin
- FileWatcher
- File Service
- VCS
- Snapshot
- Truncate

这说明它本质上是“HTTP 外壳 + 长生命周期项目实例”的运行模型，而不是每个请求临时创建上下文。

### 2.4 Agent 执行链路

一次典型任务的执行路径如下：

1. 前端在 `web/src/pages/NewTaskPage.tsx` 创建 Session。
2. 前端调用 `/session/:id/prompt_async` 异步发送任务文本。
3. 后端在 `backend/src/server/routes/session.ts` 进入 Session 域。
4. Session 模块组织消息、系统提示词、Agent 信息、工具集。
5. `backend/src/session/llm.ts` 调用 AI SDK 发起流式模型请求。
6. `backend/src/session/processor.ts` 持续消费模型流，处理：
   - reasoning delta
   - tool call
   - tool result
   - error
   - step start / finish
7. 结果被写入 Session Message / Part 持久层。
8. Bus 发布事件。
9. 前端通过 `/api/event` 订阅 SSE 并在 `TaskProcessPage.tsx` 实时渲染过程。

因此这个项目的核心不是“请求后返回一段文本”，而是“基于事件流的 Agent 过程编排”。

### 2.5 前端运行链路

前端启动入口是 `web/src/main.tsx`，路由总控在 `web/src/App.tsx`。Vite 在开发环境下把：

- `/api` 代理到 `http://localhost:4096`
- `/api/event` 代理到后端 SSE

生产环境则由 `deploy/nginx.conf` 继续承担同样的反向代理职责。

前端更像一个“Agent 控制台”，而不是业务逻辑主场。真正复杂的编排基本都在后端。

## 3. 后端项目技术层级架构

后端的技术层级可以拆成 4 层。

### 3.1 交付层（Delivery Layer）

主要文件：

- `backend/src/index.ts`
- `backend/src/server/server.ts`
- `backend/src/server/routes/*.ts`

职责：

- 进程启动
- HTTP 路由
- OpenAPI 描述
- SSE / JSON 输出
- 基础认证与中间件

这一层解决“请求如何进入系统”。

### 3.2 编排层（Orchestration Layer）

主要文件：

- `backend/src/session/processor.ts`
- `backend/src/session/llm.ts`
- `backend/src/tool/registry.ts`
- `backend/src/mcp/index.ts`
- `backend/src/project/bootstrap.ts`

职责：

- 组织模型调用
- 组装工具集
- 把权限、Provider、Session、Plugin、MCP 串起来
- 把流式结果变成结构化 Part

这一层是整个项目最关键的“大脑”。

### 3.3 领域层（Domain Layer）

主要目录：

- `backend/src/session/`
- `backend/src/provider/`
- `backend/src/tool/`
- `backend/src/permission/`
- `backend/src/project/`
- `backend/src/skill/`
- `backend/src/mcp/`
- `backend/src/schedule/`

职责：

- 维护各业务域的规则和数据结构
- 把功能按主题拆开
- 避免所有逻辑堆在 route 层

这是当前项目架构质量较高的部分，说明作者有较明确的领域分层意识。

### 3.4 基础设施层（Infrastructure Layer）

主要目录：

- `backend/src/storage/`
- `backend/src/file/`
- `backend/src/shell/`
- `backend/src/util/`

职责：

- SQLite/Drizzle 持久化
- 文件系统读写与监听
- Shell 执行
- 日志、路径、重试、上下文、懒加载等通用能力

这层负责给上层提供“可复用的运行能力”。

## 4. 后端项目目录层级架构

### 4.1 根目录结构

根目录可以简化理解为：

```text
.
├── backend/
├── web/
├── deploy/
├── docs/
├── scripts/
├── docker-compose.yml
├── Dockerfile
└── README.md
```

其中真正的核心代码集中在 `backend/` 和 `web/`。

### 4.2 backend 目录层次

`backend/src/` 是主战场，按功能域拆分，而不是按 controller/service/repository 这种传统三层切。

最重要的目录如下：

- `agent/`：Agent 定义与 Prompt 资源。
- `server/`：HTTP 服务与全部路由入口。
- `session/`：会话消息、流式处理、摘要、回滚、状态。
- `provider/`：多模型厂商适配层。
- `tool/`：内置工具与工具注册中心。
- `permission/`：权限审批机制。
- `mcp/`：MCP 客户端连接、OAuth、动态工具。
- `project/`：项目实例、工作目录、VCS、初始化。
- `storage/`：数据库与 Schema。
- `skill/`：Skill 发现与读取。
- `scheduler/` + `schedule/`：调度器与调度数据。
- `util/`：大量通用基础工具。

这种结构的好处是业务聚合清晰，但缺点是核心模块会逐步长大，特别是 `session/`、`provider/`、`tool/`。

### 4.3 web 目录层次

前端结构比较直接：

- `App.tsx`：应用壳、路由、主题、登录门禁。
- `api.ts`：所有后端接口封装与 SSE 订阅。
- `pages/`：各功能页面。
- `components/`：通用组件。
- `assets/`：静态资源。

页面级能力大致对应：

- `SessionsPage.tsx`：会话列表
- `NewTaskPage.tsx`：新建任务
- `TaskDetailPage.tsx` / `TaskProcessPage.tsx`：任务过程展示
- `DocumentsPage.tsx`：文档浏览与编辑
- `SkillsPage.tsx`：Skill 管理
- `SchedulePage.tsx`：调度任务管理
- `SettingsPage.tsx`：系统设置

前端目录结构相对轻量，没有复杂状态容器，更像操作面板。

## 5. 核心特性拆解与能力分析

### 5.1 多 Provider 大模型能力

后端在 `backend/src/provider/provider.ts` 中集成了大量 AI Provider：

- Anthropic
- OpenAI
- OpenAI Compatible
- OpenRouter
- Google / Vertex
- Bedrock
- Azure
- Mistral / Groq / Cohere / TogetherAI / Perplexity / xAI 等

能力判断：

- 不是简单单厂商接入，而是一个统一抽象层。
- 支持根据 provider 特性做参数和模型适配。
- 为未来多模型调度、替换、切换留出了空间。

这是平台型架构的重要基础。

### 5.2 Agent Tool 调用能力

`backend/src/tool/registry.ts` 是核心扩展点。当前能看到的内置工具包括：

- `bash`
- `read`
- `grep`
- `glob`
- `task`
- `todo`
- `webfetch`
- `websearch`
- `question`
- `skill`

能力判断：

- 项目本质上已经具备“工具增强型 Agent”基础能力。
- 不只是聊天，而是可以执行文件检索、Shell、并行子任务、技能调用等复杂动作。
- 工具体系支持动态扩展和插件注册，扩展性较强。

### 5.3 会话编排与过程可视化

会话系统是该项目最有价值的资产之一。

在 `backend/src/session/` 下，系统不仅保存消息，还把 Agent 的过程拆成 Part：

- text
- reasoning
- tool
- step-start
- step-finish

前端 `TaskProcessPage.tsx` 再把这些 Part 还原成过程视图。

能力判断：

- 这已经不是普通“聊天记录”模型，而是“过程型 Agent UI”。
- 对调试、解释、审计、回放都很有价值。
- 如果后续增加更多结构化 Part 类型，平台表达能力还会继续增强。

### 5.4 权限控制能力

`backend/src/permission/service.ts` 提供了按工具、按路径模式、按 reply 结果的审批机制。

能力判断：

- 说明项目意识到 Agent 工具调用的风险，不是无条件执行。
- 已具备 allow / deny / ask 三种决策模型。
- 还支持规则持久化。

这对 Shell / 文件系统 / 外部目录访问类工具尤为关键。

### 5.5 MCP 扩展能力

MCP 模块位于 `backend/src/mcp/`，已具备：

- 多传输方式连接
- OAuth 授权
- 动态工具转换
- prompt / resource 拉取

能力判断：

- 这意味着平台不是封闭型 Agent，而是可以把外部能力通过 MCP 注入。
- 如果后续接更多 MCP Server，它的外部生态扩展能力会明显增强。

这是该项目非常重要的“平台化”能力。

### 5.6 文档工作区能力

前端 `DocumentsPage.tsx` 提供了一个项目内文档浏览与编辑界面，并集成 Monaco 编辑器与 Markdown 预览。

能力判断：

- 这让平台不只是“运行任务”，还能回到文档资产管理场景。
- 很适合和 Agent 输出、Skill 文档、计划文档联动。

### 5.7 调度能力

当前已存在调度页面和 Schedule API。

后端 `backend/src/server/routes/schedule.ts` + `backend/src/scheduler/index.ts` 提供：

- 创建调度
- 更新调度
- 删除调度
- 手动触发调度

能力判断：

- 当前更接近 MVP 能力。
- 已完成产品闭环雏形，但 cron 解析和执行语义还比较初级。
- 后续若增强，可演进成自动化 Agent 任务平台。

## 6. 项目核心优势

### 6.1 架构方向是对的

当前代码清楚地表明，这个项目已经走在“Agent 平台化”方向上，而不是停留在 Demo 级聊天应用：

- 有 Provider 抽象
- 有 Tool Registry
- 有 Session Runtime
- 有 Permission System
- 有 MCP Integration
- 有 SSE 实时过程流

这几个点叠加起来，说明项目骨架是成熟方向。

### 6.2 后端组织质量高于一般原型项目

虽然模块很多，但整体并不是杂乱堆砌。特别是：

- `session`
- `provider`
- `tool`
- `project`
- `permission`
- `mcp`

这些域划分是清楚的，说明代码作者对复杂系统拆解有经验。

### 6.3 前端目标明确

前端没有过度承载复杂业务，而是专注于：

- 会话与任务入口
- 过程可视化
- 文档操作
- Skill / Settings / Schedule 管理

这让系统复杂度集中在后端，更适合 Agent 类产品。

## 7. 当前主要问题与风险

### 7.1 测试体系明显缺失

当前几乎没有成型的一方测试代码。虽然有 TypeScript 检查和前端 lint，但缺少：

- 单元测试
- 集成测试
- SSE 流式回归测试
- Tool 执行链路测试
- Provider 适配测试

这会导致系统越复杂，后续越难安全迭代。

### 7.2 安全基线依赖环境变量

后端只有在设置 `OPENCODE_SERVER_PASSWORD` 时才启用 Basic Auth。对一个能执行 Shell / 文件 / 外部工具的系统来说，这个默认风险偏高。

### 7.3 若干平台能力仍是占位实现

`config.ts` 与 `provider.ts` 中明显存在 Auth / Account / Installation 等 stub，这说明当前仓库并不完全等于最终生产态，而是处于改造或迁移阶段。

这类代码会给后续接手者造成误判：看起来有能力，实际上可能尚未打通。

### 7.4 调度系统尚未产品化完成

目前调度更像功能雏形：

- cron 支持有限
- in-memory interval 调度较轻量
- 缺少更强的失败重试、精确计划、持久执行语义

如果未来调度要成为卖点，需要单独升级。

### 7.5 核心模块未来会变得很重

现在已经能看到几个高复杂度中心：

- `session/processor.ts`
- `provider/provider.ts`
- `session/index.ts`
- `mcp/index.ts`

这些模块会随着功能继续膨胀，后续如果不做进一步分裂，会成为维护瓶颈。

---

## 8. 结论

这个项目当前最准确的定位是：

**一个以后端 Agent Runtime 为核心、前端控制台为外壳、具备多 Provider / 多工具 / MCP / 实时过程可视化能力的 Web AI Agent 平台。**

如果从“项目拆解”的角度总结，它的核心骨架可以归纳为 3 句话：

1. 后端是平台核心，承担了 Agent 的绝大多数复杂性。
2. 前端是操作台，负责把 Session、Task、Document、Skill、Schedule 可视化。
3. 真正的战略价值在于统一了 Provider、Tool、Permission、MCP、Session Runtime 这几条能力链。

从成熟度看：

- 架构方向：较好
- 核心能力骨架：完整
- 工程验证体系：偏弱
- 安全与产品化细节：仍需加强

如果后续继续深挖，最值得优先投入的方向不是“再加几个页面”，而是：

- 增强测试与 CI
- 收敛 stub/迁移代码
- 强化权限与默认安全策略
- 对 Session / Provider / MCP 做进一步可维护性拆分

这会直接决定它能否从“结构很强的原型平台”走向“可持续演进的生产级 Agent 系统”。
